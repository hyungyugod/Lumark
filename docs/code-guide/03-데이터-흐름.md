# Lumark 코드 구조, 중학생 버전 (3) — 데이터가 흐르는 길

> 2편이 "공장 부서별 사무실"이었다면, 3편은 **재료가 들어와서 완성품이 나가기까지의 컨베이어 벨트**.
> "굿노트 PDF가 어떻게 마크다운 노트가 되나"를 6단계로 풀어 봤어.

---

## 🚪 입구 3가지 — PDF가 들어오는 길

Lumark는 PDF/사진이 **3가지 길**로 들어올 수 있어.

### 길 1: 직접 골라서 넣기

```
[사용자가 홈 화면에서 "PDF 가져오기" 누름]
        ↓
[HomeView.swift]   → PhotosPicker / fileImporter 띄움
        ↓
[사용자가 PDF 선택]
        ↓
[HomeView가 startProcessing 호출]
        ↓
[ProcessingView로 이동]
```

### 길 2: 카메라로 찍기

```
[사용자가 "카메라" 누름]
        ↓
[DocumentScannerView.swift]   → iOS 기본 스캐너 띄움
        ↓
[사진 여러 장 찍음]
        ↓
[HomeView가 ingestScannedImages 호출]
        ↓
[ProcessingView로 이동]
```

### 길 3: 굿노트에서 공유 (별관 입구)

```
[굿노트 앱에서 PDF 공유 누름]
        ↓
[ShareViewController.swift]   → 공유 시트 UI 띄움
        ↓
[사용자가 "변환 시작" 누름]
        ↓
[AppGroup.stage(...)]   → 공용 폴더(group.com.lumark/Inbox)에 PDF 저장
        ↓
[lumark://import?id=xxx 링크로 메인 앱 깨움]
        ↓
[LumarkApp.onOpenURL → AppRouter.handle]
        ↓
[HomeView가 ingestInbox 호출]
        ↓
[ProcessingView로 이동]
```

> 3가지 길이 다르게 시작해도 **결국 ProcessingView 앞에서 모두 합류**해.
> 그 다음부터는 똑같은 컨베이어 벨트를 탐.

---

## ⚙️ 컨베이어 벨트 — 6단계 처리 파이프라인

PDF가 ProcessingView에 도착한 다음의 흐름:

```
[1] PageRenderer
    PDF를 페이지별 이미지로 쪼개기
        ↓
[2] HighlightDetector  ⚠️ 아직 미구현 (Mock 동작)
    이미지에서 형광펜 색 부분만 골라내기
        ↓
[3] OCRService  ⚠️ 아직 미구현 (Mock 동작)
    형광펜 친 부분의 글자 읽기
        ↓
[4] 색별 그룹핑 + 정렬
    같은 색끼리 묶고 페이지 순서대로 정렬
        ↓
[5] 구조 인식 (splitByOrange)
    "주황은 섹션 제목" 규칙으로 마크다운 구조 만들기
        ↓
[6] 출력
    ├ MarkdownExporter → .md 텍스트
    └ PDFExporter      → PDF 파일
```

### 각 단계 자세히

**[1] PageRenderer**
- 파일: `Services/PageRenderer.swift`
- 하는 일: PDF를 받아서 "1페이지 이미지, 2페이지 이미지, ..." 처럼 한 장씩 분리.
- 빈 페이지는 자동으로 건너뜀.

**[2] HighlightDetector ⚠️**
- 아직 코드로 안 만들어졌어 (Day 2~4 작업 예정).
- 만들어지면: 이미지 속 노랑/주황/분홍/파랑 픽셀만 골라내서 "이 부분이 형광펜이다"라고 마킹.
- 기술: Core Image의 HSV(색·채도·명도) 마스킹.

**[3] OCRService ⚠️**
- 역시 아직 미구현.
- 만들어지면: 형광펜 친 부분만 잘라서 Apple Vision Framework로 글자 인식.
- 한국어 인쇄체에 특화.

**[4] 색별 그룹핑**
- 파일: `Services/MarkdownDocument.swift` 안의 `MarkdownDocument.from(Note)` 함수.
- 노랑은 노랑끼리, 주황은 주황끼리 묶고, 페이지·순서대로 정렬.

**[5] 구조 인식 (splitByOrange)**
- 같은 파일 안의 알고리즘.
- 규칙: **"주황이 나오면 새 섹션 시작, 노랑들은 그 섹션의 본문"**
- 예: 주황 "항생제" → 노랑 "페니실린, 세팔로스포린, ..." → 주황 "부작용" → 노랑 "...".
- 이게 곧 마크다운의 `## 제목` + 그 아래 `- 본문`이 됨.

**[6] 출력**
- `MarkdownExporter.swift` → CommonMark 또는 Obsidian 문법 텍스트 생성.
- `PDFExporter.swift` → CoreText로 페이지 분할된 PDF 인쇄.

> **지금(2026-05) 상태**: [1], [4], [5], [6]은 작동. [2], [3]은 Mock(가짜 타이머)로 동작.
> 그래서 지금 변환을 돌리면 결과는 항상 `MockData.antibioticsNote()` (가짜 항생제 노트)가 나옴.
> 실제 OCR을 끼우기 전 단계.

## 🗃️ 결과는 어디 저장돼?

만들어진 노트는 **SwiftData**라는 iOS 내장 데이터베이스에 들어가.

```
SwiftData (앱 안 데이터베이스)
└─ Note ─── (1:N) ─── Page ─── (1:N) ─── Highlight
      │
      ├─ id, 제목, 만든 날짜, 페이지 수, 즐겨찾기 여부
      └─ 페이지 이미지는 따로 외부 저장(externalStorage)
         → DB가 무거워지지 않게
```

추가로 작은 설정들은 **UserDefaults**에 저장:
- `com.lumark.colorRules` — 색깔 규칙 (노랑=본문 등)
- `lumark.export.dialect` — 마크다운 스타일 선택
- `lumark.onboarded` — 처음 켰는지 플래그

진행 중이던 작업은 **lumark.jobs.json**에 따로 저장 → 앱이 꺼졌다 켜져도 이어서 가능.

## 🚦 화면 라우팅 — 어디서 어디로 가는가

```
HomeView (홈)
   │
   ├─ "PDF/사진 가져오기" → ProcessingView → ResultView
   │
   ├─ "카메라" → DocumentScannerView → ProcessingView → ResultView
   │
   ├─ "최근 노트" → RecentNotesView → ResultView (탭한 노트)
   │
   ├─ "설정" → SettingsView (sheet으로 위에서 올라옴)
   │
   └─ 첫 실행 시 → OnboardingSheet
```

규칙:
- **새 변환 완료**: `path = [.result(id)]` — 처리 화면을 지우고 결과로 교체. 뒤로 가기 누르면 홈.
- **기존 노트 탭**: `path.append(.result(id))` — 누적. 뒤로 가기 누르면 최근 노트 리스트.

이 차이를 알면 "왜 어떤 때는 뒤로 가면 홈인데 어떤 때는 리스트로 가지?"가 풀려.

## 🎨 색깔 → 마크다운 매핑 규칙 (이 앱의 핵심 아이디어)

| 형광펜 색 | 기본 의미 | 마크다운 출력 |
|---|---|---|
| 🟠 주황 | 섹션 제목 | `## 제목` |
| 🟡 노랑 | 핵심 본문 | `- 글머리표` |
| 🔵 파랑 | 추가 메모 (옵션) | 별도 메모 영역 |
| 🌸 분홍 | 추가 메모 (옵션) | 별도 메모 영역 |

이 규칙은 사용자가 설정에서 **바꿀 수도 있어** (`ColorRuleStore.swift`).
예: "내 주황은 사실 인용문이야"로 바꾸면 마크다운이 그렇게 나옴.

## ⚠️ 에러는 어떻게 다뤄?

`Services/LumarkError.swift`가 에러 14종을 정의. 각 에러는 4가지 정보를 들고 있어:

1. **debugCode** — 개발자용 코드 (예: `"OCR-EMPTY"`, `"AG-01"`)
2. **userTitle / userMessage** — 사용자에게 보일 친절한 문구
3. **defaultActions** — 다음에 할 수 있는 행동 (절대 비어 있으면 안 됨 = **막다른 길 금지**)
4. **severity** — 심각도 (error / warning / info)

> 핵심 디자인 철학: **"사용자가 막다른 골목에 갇히면 안 된다."**
> 모든 에러는 "다시 시도", "홈으로", "다른 파일 선택" 같은 출구를 최소 1개 제공.

## 🏃 동시성 (백그라운드 처리)

이건 좀 어렵지만 핵심만:

- **기본 규칙**: Swift는 화면 그리는 작업을 메인 스레드에서만 함 (안 그러면 앱이 느려져).
- **하지만 OCR이나 PDF 렌더링은 무거운 작업** → 백그라운드에서 돌아야 함.
- 그래서 무거운 일꾼들(`PageRenderer`, `PDFExporter`, `MarkdownDocument`)은 코드에 `nonisolated` 표시 → "나는 메인 스레드 아니어도 돼"라고 선언.

> 비유: 식당에서 손님 받는 직원(메인)이 직접 요리(무거운 작업) 하면 줄이 늘어남.
> 그래서 주방 셰프(백그라운드)한테 보내고, 메인 직원은 다른 손님 응대.
