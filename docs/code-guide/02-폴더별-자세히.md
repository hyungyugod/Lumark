# Lumark 코드 구조, 중학생 버전 (2) — 폴더별 자세히

> 1편이 "공장 전체 도면"이었다면, 2편은 **각 부서 사무실에 들어가서 책상 하나하나 본다**.
> 폴더 → 파일 순서로, 각 파일이 무슨 일을 하는지 한두 줄로 정리했어.

---

## 📂 `Lumark/App/` — 정문 안내데스크

앱이 켜졌을 때 가장 먼저 들어오는 곳. 3개 파일밖에 없음.

| 파일 | 한 줄 역할 |
|---|---|
| `LumarkApp.swift` | iOS가 "앱 켜!"라고 했을 때 진짜 시작점. 어디 화면 띄울지, 데이터베이스(SwiftData) 어디 둘지 세팅. |
| `AppRouting.swift` | "어디 어디로 갈 수 있다"는 길의 종류만 정의한 **지도** (열거형). |
| `AppRouter.swift` | 외부에서 들어온 신호(예: `lumark://import?id=...` 링크)를 보고 **어디로 보낼지 결정**하는 안내원. |

> **핵심 비유**: 백화점 정문 안내데스크. "1층 화장품관 가실 거면 이쪽요"를 결정.

---

## 📂 `Lumark/Models/` — 빈 박스 설계도

데이터의 **모양**만 정의한 곳. 실제로 일하지는 않아.

| 파일 | 한 줄 역할 |
|---|---|
| `Note.swift` | "노트 한 권"의 모양 — 제목, 만든 날짜, 페이지 수, 즐겨찾기 여부 등. |
| `Page.swift` | "페이지 한 장"의 모양 — 페이지 번호, 페이지 이미지 데이터. |
| `Highlight.swift` | "형광펜 한 줄"의 모양 — 색깔, OCR로 읽은 글자, 페이지에서의 위치. |
| `ColorCategory.swift` | "색깔 카테고리" 정의 — 노랑/주황/분홍/파랑 같은 분류. |
| `ColorRule.swift` | "이 색깔은 이런 의미"라는 사용자 규칙 (예: 노랑 = 본문). |
| `MockData.swift` | OCR이 아직 안 만들어졌을 때 쓰는 **가짜 노트 데이터** (테스트용). |

> **핵심 비유**: 가구 매장에서 "이게 책장이고, 책장 안에 칸이 있고, 칸 안에 책이 있어요"라고 보여주는 **샘플 가구**.
> 진짜 책장이 아니라 "이런 모양이다"를 정의한 종이일 뿐.

## 📂 `Lumark/Services/` — 진짜 일꾼들

여기가 **앱의 두뇌**야. 실제로 OCR 돌리고, 마크다운 만들고, PDF 뽑는 곳.

| 파일 | 한 줄 역할 |
|---|---|
| `PageRenderer.swift` | PDF를 한 장씩 이미지로 변환. "PDF 4페이지짜리" → "이미지 4장"으로 풀어줌. |
| `MarkdownDocument.swift` | 노트 데이터를 받아서 **마크다운 구조**(제목·본문·메모)로 변환. **이 앱의 알고리즘 심장**. |
| `MarkdownExporter.swift` | 마크다운 구조를 진짜 `.md` 파일 텍스트로 출력. |
| `PDFExporter.swift` | 같은 내용을 보기 좋은 PDF로 인쇄. |
| `AppGroup.swift` | 메인 앱과 Share Extension(별관)이 **같은 폴더를 공유**할 수 있게 해주는 통로. |
| `URLSchemeRouter.swift` | `lumark://import?id=...` 같은 특수 링크를 해석. |
| `PermissionService.swift` | 카메라 권한 같은 거 물어보고 받기. |
| `JobStateStore.swift` | "변환 중이던 작업"을 저장해서, 앱이 꺼졌다 켜져도 이어서 할 수 있게. |
| `LumarkError.swift` | 에러 14종류 정의 + 사용자한테 보여줄 친절한 메시지 묶음. |
| `ExportPreferences.swift` | "마크다운을 CommonMark 스타일로 뽑을까, Obsidian 스타일로 뽑을까" 같은 사용자 선택 저장. |

> **핵심 비유**: 식당 주방. 손님(Views)이 주문하면 셰프(Services)가 진짜 요리(작업)를 만들어.
> 손님은 주방이 어떻게 굴러가는지 몰라도 돼.

**Services가 너무 많아서 헷갈리면** 이렇게 3그룹으로 묶어봐:

1. **변환 일꾼**: `PageRenderer`, `MarkdownDocument`, `MarkdownExporter`, `PDFExporter`
2. **시스템 연결**: `AppGroup`, `URLSchemeRouter`, `PermissionService`
3. **상태 보관·에러**: `JobStateStore`, `LumarkError`, `ExportPreferences`

## 📂 `Lumark/ViewModels/` — 비서

화면(Views)이 일꾼(Services)한테 직접 말 걸지 않게, **사이에 끼는 통역사**.

| 파일 | 한 줄 역할 |
|---|---|
| `ProcessingViewModel.swift` | "변환 중" 화면 뒤에서 진행률·상태를 관리. Service들 호출해서 결과를 화면에 넘김. |

> ViewModel이 단 1개인 이유? — 다른 화면들은 단순해서 ViewModel 없이도 굴러가게 짠 것.
> 복잡한 처리 화면 하나만 비서가 필요했음.

---

## 📂 `Lumark/Views/` — 사용자가 보는 화면

진열대. 여기가 사용자가 손가락으로 누르는 곳.

### 메인 화면들

| 파일 | 한 줄 역할 |
|---|---|
| `HomeView.swift` | 앱 첫 화면. PDF 고르기, 카메라, 최근 노트 보기 같은 액션 카드들. (669줄, 가장 큼) |
| `ProcessingView.swift` | "변환 중..." 진행률 화면. |
| `ResultView.swift` | 변환 결과 화면. 마크다운 보기 ↔ 원본 PDF 보기 탭, 색깔 필터, 공유 버튼. |
| `RecentNotesView.swift` | 지금까지 만든 노트들 리스트. 검색·정렬·즐겨찾기. |
| `SettingsView.swift` | 설정 화면. 색깔 규칙 바꾸기, 마크다운 스타일 선택 등. |
| `OnboardingSheet.swift` | 앱 처음 켤 때 "이 앱 이렇게 써요" 안내. |

| 파일 | 한 줄 역할 |

| `DocumentScannerView.swift` | iOS 기본 카메라 스캐너 띄우는 래퍼. |
| `EmptyStateView.swift` | "아직 노트가 없어요" 같은 빈 상태 안내. |
| `ErrorView.swift` | 에러 났을 때 풀스크린/배너로 보여주는 화면. |

> 같은 소품(예: ActionCard)을 여러 화면에서 다시 갖다 쓸 수 있어서 코드 중복이 줄어.

## 📂 `Lumark/ShareExtension/` — 별관

이건 사실 **메인 앱과 별개로 돌아가는 작은 미니 앱**이야. iOS에서 굿노트 같은 다른 앱이 "공유" 누를 때 뜨는 시트.

| 파일 | 한 줄 역할 |
|---|---|
| `ShareViewController.swift` | iOS가 공유 요청을 보내면 가장 먼저 받는 컨트롤러. |
| `ShareView.swift` | 공유 시트 위에 뜨는 미리보기 + "변환 시작" 버튼 UI. |

흐름:
1. 굿노트에서 PDF를 공유 → ShareExtension이 받음
2. PDF를 **공용 폴더(App Group)** 에 저장
3. `lumark://import?id=xxx` 링크로 메인 앱을 깨움
4. 메인 앱이 그 폴더에서 PDF를 집어와서 처리 시작

## 정리 한 줄

> **`Lumark/App/`은 정문, `Lumark/Models/`는 빈 박스, `Lumark/Services/`는 일꾼,
> `Lumark/Views/`는 진열대, `Lumark/ShareExtension/`은 별관, 나머지는 보조.**
> 이 6개만 외워두면 코드 어디를 봐도 안 길을 잃어.
