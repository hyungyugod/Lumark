# Lumark 코드 구조, 중학생 버전 (1) — 전체를 한눈에

> 이 폴더(`docs/code-guide/`)는 코드 한 줄도 모르는 사람도 "이 앱이 어떻게 만들어졌구나"를 그림처럼 이해할 수 있게 쓴 설명서야.
> 1편은 **숲(전체 구조)**, 2편은 **나무(폴더별 역할)**, 3편은 **줄기(데이터가 흘러가는 길)** 를 본다.

---

## 1. Lumark가 도대체 뭐 하는 앱인데?

한 줄 요약:
> **"형광펜만 그으면, 정리 노트가 알아서 쌓이는 iOS 앱."**

이걸 풀어 쓰면 이런 흐름이야.

1. 너는 굿노트 PDF나 사진에 **형광펜**으로 그어. (노랑 = 핵심, 주황 = 섹션 제목, 분홍·파랑 = 메모)
2. 그걸 Lumark 앱에 넣어줘. (직접 열거나, 굿노트에서 "공유" 누르거나, 카메라로 찍거나)
3. 앱이 **사진 속 형광펜 색을 인식** → **글자를 읽음(OCR)** → **마크다운 노트**로 정리해줘.
4. 너는 마크다운 또는 PDF로 받거나, 앱 안에 노트 라이브러리처럼 쌓아둬.

핵심 아이디어 한 줄:

> **"색깔 = 의미"**
> 그래서 어떤 색을 어떤 의미로 쓸지 정해두면(예: 노랑 = 본문, 주황 = 제목),
> 앱이 그 규칙대로 마크다운 구조를 만들어줘.

---

## 2. 이 앱을 "공장"이라고 상상하면 좋아

코드가 그냥 한 덩어리로 있는 게 아니라, **공장처럼 부서가 나뉘어** 있어.

| 공장 비유 | 코드 폴더 | 하는 일 |
|---|---|---|
| 정문 / 안내데스크 | `Lumark/App/` | 앱이 켜질 때 어디로 보낼지 결정 |
| 진열대 (사용자가 보는 화면) | `Lumark/Views/` | 홈, 처리 중, 결과, 설정 같은 화면들 |
| 비서 (화면과 일꾼 사이) | `Lumark/ViewModels/` | 화면이 시킨 일을 일꾼들한테 전달 |
| 일꾼들 (실제로 일하는 사람) | `Lumark/Services/` | OCR, 마크다운 만들기, PDF 뽑기 같은 진짜 작업 |
| 설계도 / 빈 박스 | `Lumark/Models/` | "노트는 이렇게 생긴 박스다"라고 정의해둔 종이 |
| 창고 (설정 보관) | `Lumark/Repositories/` | "이 색깔을 어떻게 쓸지" 같은 사용자 설정 보관 |
| 인테리어 | `Lumark/Theme/` | 앱 전체 색깔·폰트·디자인 톤 |
| 별관 (옆 건물) | `Lumark/ShareExtension/` | 굿노트에서 "공유"로 들어오는 작은 입구 |
| 검수실 | `LumarkTests/`, `LumarkUITests/` | "공장이 제대로 돌아가나" 자동으로 확인 |

> 이 표만 외워두면, 어떤 코드 파일을 봐도 **"이건 어느 부서 직원이지?"** 를 바로 답할 수 있어.

---

## 3. 부서끼리 서로 어떻게 부르는데? (호출 규칙)

부서는 **위에서 아래로만** 말을 걸 수 있어. 이걸 "레이어 아키텍처"라고 해.

```
┌─────────────────────────────────────────┐
│ Views        화면 (사용자가 누름)         │ ← 사용자가 여기 만짐
├─────────────────────────────────────────┤
│ ViewModels   비서 (화면이 시킬 때 통역)    │
├─────────────────────────────────────────┤
│ Services     일꾼 (실제로 OCR·PDF 작업)   │
├─────────────────────────────────────────┤
│ Repositories 창고 (설정 보관)             │
├─────────────────────────────────────────┤
│ Models       빈 박스 (데이터 모양)         │
└─────────────────────────────────────────┘
```

**중요한 규칙 3가지:**

1. **위가 아래를 부른다.** Views → ViewModels → Services → Models 방향만 OK.
2. **아래는 위를 모른다.** Models는 "내가 어디서 쓰이는지" 모른 채 그냥 자기 모양만 정의해.
3. **Views는 Models를 직접 만지지 않는다.** 꼭 ViewModel이나 Service라는 다리를 거쳐.

이렇게 나누는 이유는 **"한 군데 고장 나도 다른 데가 안 무너지게"** 하기 위해서야.
예를 들어 OCR 라이브러리를 바꿔도 Views는 건드릴 필요가 없어. Services 안에서만 바꾸면 끝.

---

## 4. 데이터 박스(Model)는 딱 3개만 기억해

앱이 다루는 핵심 데이터는 결국 이 셋이야.

```
Note  ─── 안에 ──→  Page  ─── 안에 ──→  Highlight
(노트)              (페이지)            (형광펜 칠한 한 조각)
```

- **Note** = 노트 한 권. 예: "5월 25일 약리학 스캔"
- **Page** = 그 노트 안의 한 페이지. 사진 한 장 = 한 Page.
- **Highlight** = 페이지 안의 형광펜 한 줄. "노랑이고, OCR 결과는 '아미노글리코사이드'"

> 마치 **공책 한 권 → 페이지 → 페이지 안의 한 줄** 처럼 들어가 있다고 보면 돼.

이걸 코드에서는 `Lumark/Models/Note.swift`, `Page.swift`, `Highlight.swift` 세 파일이 정의해.
